# 產品需求文件與開發流程

> 當你需要撰寫 PRD、與工程師溝通、或了解軟體開發流程時，使用這份指引。

---

## 1. 產品需求文件（PRD）

### PRD 的角色

PRD 是設計部門與其他部門（特別是工程團隊）**精準溝通的橋樑**。

目的不是「寫很多字」，而是讓沒參與設計過程的人也能完全理解產品設計的意圖。

### PRD 與其他文件的關係

| 文件 | 聚焦 | 對象 |
|---|---|---|
| 商業需求文件（BRD）| 商業可行性、策略、財務 | 高層決策 |
| 市場需求文件（MRD）| 目標族群、競爭、行銷規劃 | 行銷團隊 |
| 產品需求文件（PRD）| 功能規格、互動細節、技術需求 | 工程團隊 |

### PRD 的組成要素

#### 1. 檔名與版本管理
- 文件版本號 + 日期
- 修改記錄（誰改了什麼、什麼時候）
- 負責人（PM、設計師、工程師、測試）

#### 2. 產品背景
- 產品理念、願景、目標
- 想要解決的問題 / 痛點
- 使用情境描述（可用故事情節模擬）

#### 3. 目標族群與需求
- Persona 描述
- 客戶旅程地圖
- 關鍵需求列表
- 訪談原音重現（選填，但很有說服力）

#### 4. 功能需求與價值說明
- 功能列表
- 每個功能的 User Story
- 功能對使用者的好處 / 價值
- 商業機會標註
- 標記 MVP 關鍵功能

#### 5. 功能架構與流程圖
- 系統模組架構圖
- 服務互動流程圖
- 讓工程師理解「全貌」

#### 6. 介面原型與互動流程
- Wireframe / Wireflow
- 可互動的 Prototype 連結
- 頁面間的邏輯關係

#### 7. 系統與數據規格（選填但加分）
- 系統運作邏輯圖
- 資料庫欄位定義
- API 規格
- 不需要精通，但有基本概念 → 和工程師溝通更順暢

#### 8. 成功指標
- 功能成功的衡量指標
- 需要收集的數據
- 數據呈現方式

#### 9. 非功能需求
- 品牌規範（配合合作方的 CIS）
- 效能需求（同時在線人數）
- 多語言 / 在地化需求
- 平台相容性（iOS / Android / Web）

#### 10. 已知風險
- 法規風險（金融法規、個資法）
- 技術風險（AI 技術成熟度、第三方依賴）
- 資安風險（數據保護）

### PRD 最佳實踐

- 使用具有共編功能的工具（Google Docs、Notion、Gitbook）
- 文件要能**不需要任何人解釋，閱讀者就能看懂**
- 支援版本追蹤和修改記錄
- 支援遠端同步即時溝通
- 支援段落級別的註記和討論
- 啟用通知功能

### PRD 不是一次性文件

它是跟著產品研發過程持續更新的活文件。所有和產品有關的人都可以在上面補充想法。

## 2. 設計與工程的溝通

### 設計師需要理解的工程概念

不需要會寫程式，但需要了解：

- **前端 vs 後端**：介面是前端，邏輯和數據是後端
- **API**：系統間溝通的介面
- **資料庫基本概念**：數據存在哪裡、欄位是什麼
- **技術限制**：什麼在技術上做不到 / 很難做

### 溝通原則

1. **設計的意圖比設計的外觀重要**：告訴工程師「為什麼這樣設計」而非只給圖
2. **歡迎工程師的替代方案**：工程師可能有更好的技術解決方案
3. **共同討論可行性**：設計規格書不是命令，是討論的起點
4. **產品願景要分享**：讓工程師知道未來方向，有助於技術架構規劃
5. **微型互動要確認可行性**：動畫效果、過場動畫等在實作上的成本

## 3. 軟體開發流程

### 瀑布式開發（Waterfall）

需求 → 設計 → 開發 → 測試 → 上線

- 線性流程，每階段完成才進下一階段
- 適合需求非常明確、不太會變的專案
- 缺點：不夠靈活，發現問題時修改成本高

### 敏捷式開發（Agile）

- 短週期迭代（Sprint，通常 1-2 週）
- 每個 Sprint 交付可用的功能增量
- 持續收集回饋、持續調整方向

### Scrum 框架

- **Product Backlog**：所有待做的功能列表
- **Sprint Planning**：選出這個 Sprint 要做的功能
- **Daily Standup**：每天 15 分鐘同步進度
- **Sprint Review**：展示 Sprint 成果
- **Sprint Retrospective**：回顧什麼做得好、什麼要改善

### UX 設計師在敏捷中的角色

- 設計需要**跑在開發前面** 1-2 個 Sprint
- 設計師需要與開發團隊保持密切溝通
- 設計並非在開發前全部完成 — 設計和開發是並行的
- 設計師要參與 Sprint Review，確認實作符合設計意圖

### 設計與開發的協作模式

```
Sprint N:    設計師設計功能 B + C
             工程師開發功能 A（上個 Sprint 設計完成的）
Sprint N+1:  設計師設計功能 D + E
             工程師開發功能 B + C
             設計師驗收功能 A 的實作品質
```

設計永遠比開發超前一點，但也要回頭驗收。
