# 互動設計與介面設計

> 當你在做資訊架構、互動流程、介面設計，或被打槍不知道怎麼改時，使用這份指引。

---

## 1. 人機互動設計（HCI）

### 核心觀念

人與電腦的溝通，就像兩個人對話：
- 系統說什麼（資訊呈現）
- 使用者回應什麼（操作輸入）
- 系統再回應什麼（回饋）

### 互動設計的決策層次

1. **互動裝置選擇**：
   - 視覺 + 觸控（手機、平板）
   - 聽覺 + 語音（智慧音箱）
   - 手勢 / 體感（VR/AR）
   - 每種方式的能力與限制不同

2. **互動主導權**：
   - 系統主動：系統替使用者做決定（自動推薦）
   - 系統被動：提供選項讓使用者選擇（選單）
   - 使用者主動：使用者自由操作，系統提供回饋（搜尋）

3. **回饋設計**：
   - 每個操作都需要回饋（讓使用者知道系統接收到了）
   - 回饋要即時、明確、合適

## 2. 資訊架構（Information Architecture）

### 什麼是資訊架構？

用人們理解世界的方式來組織系統中的資訊。

### 四個核心問題

1. **資訊如何分類？** — 分類方式要符合使用者認知，而非開發者邏輯
2. **資訊如何命名？** — 用使用者的語言，不是行業術語
3. **資訊如何呈現？** — 層級、權重、視覺區分
4. **資訊如何被找到？** — 導覽、搜尋、麵包屑

### 設計原則

- **符合心智模型**：資訊的分類、命名、位置要符合使用者的既有認知
- **漸進式揭露**：不要一次展示所有資訊，按需展開
- **一致性**：相同的功能在不同頁面有相同的位置和表現

### 功能架構圖

- 用結構化的方式呈現整個產品的功能全貌
- 讓工程師理解：有哪些模組、模組間什麼關係、資訊如何流動
- 幫助團隊用「一張圖」對齊產品理解

## 3. 使用者介面設計（UI Design）

### 介面設計不是美工

UI 設計師的專業是：在每個互動狀態下，提供使用者能理解、可操作的介面元素，讓使用者輕鬆駕馭產品。

### UI 設計流程

1. **Wireframe（線框圖）**：低保真，確認佈局和功能位置
2. **Wireflow**：Wireframe + 頁面間的流程邏輯
3. **Visual Design（視覺設計）**：高保真，確認色彩、字型、風格
4. **Prototype（可互動原型）**：可以點擊操作的模擬
5. **Design System（設計系統）**：統一的元件庫和規範

### 介面設計核心原則

**一致性原則**：
- 相同功能 = 相同外觀和行為
- 跨頁面的控制元件保持統一
- 符合平台規範（iOS / Android / Web）

**簡約原則**：
- 每個畫面只聚焦一個主要任務
- 減少視覺雜訊
- 關鍵操作要顯眼，次要操作可收合

**可見性原則**：
- 可操作的元素看起來要「可以操作」
- 狀態變化要有視覺回饋
- 重要資訊不能藏在深層選單

**容錯原則**：
- 防止操作錯誤（確認對話框、灰顯不可用選項）
- 提供撤銷機制
- 錯誤訊息要說清楚「什麼錯了」和「怎麼修」

**操作動線原則**：
- 按鈕位置不是靠「感覺」— 是根據操作流程動線決定
- 人的視線遵循特定模式（F型、Z型閱讀）
- 拇指熱區：手機介面的關鍵操作要在拇指可觸及範圍

### 控制元件選擇

- 選項少（2-5 個）→ 直接列出（Radio Button / Toggle）
- 選項多（5+）→ 下拉選單
- 是/否決定 → 開關（Switch）
- 範圍選擇 → 滑桿（Slider）
- 自由輸入 → 文字框（考慮：要不要加自動完成？）

## 4. 原型設計與測試

### 原型的角色

原型是「用最低成本驗證設計假設」的工具。

### 原型層次

- **紙上原型**：手繪草圖，5 分鐘完成，適合早期發想
- **低保真數位原型**：Wireframe 層級，確認功能和流程
- **高保真原型**：接近真實產品，適合使用性測試
- **可互動原型**：可點擊操作，適合讓使用者實際體驗

### 原型測試方法

1. **認知走查（Cognitive Walkthrough）**：
   - 專家模擬使用者操作流程
   - 檢查每一步是否符合使用者預期

2. **使用性測試（Usability Test）**：
   - 找真實使用者操作原型
   - 觀察他們在哪裡卡關、困惑、出錯
   - 5 位使用者可以發現約 85% 的問題

3. **A/B 測試**：
   - 兩個版本給不同使用者群
   - 用數據比較哪個版本效果更好

### 測試後的處理

- 使用者的回饋需要**整理和消化**，不是照單全收
- 需要透過設計評論會議，結合專業判斷做取捨
- 並非所有使用者回饋都應該被實現

## 5. 設計評論（Design Critique）

### 什麼是設計評論？

與團隊和專家一起，系統性地評估設計方案的品質。

### 正確做法

- 基於**原則和目標**評價，而非個人喜好
- 「這個按鈕放右邊比較好」→ 不可接受
- 「根據 F 型閱讀動線，按鈕放在視線終點比較容易被發現」→ 可接受

### 評論框架

1. 這個設計想解決什麼問題？
2. 目標使用者是誰？
3. 這個方案如何解決問題？
4. 有什麼優點？
5. 有什麼可以改善的？（提出替代方案）
6. 數據上有支持嗎？

## 6. 設計資源

### UI 設計資源

- 設計規範：Material Design（Android）、Human Interface Guidelines（iOS）
- 元件庫：參考成熟的設計系統
- 圖示資源：Icon 庫
- 配色工具：色彩搭配工具
- 設計師作品：Dribbble、Behance 等社群

### 使用資源的原則

- 參考不是抄襲 — 理解背後的設計邏輯
- 在形成自己的設計美感和直覺之前，大量參考是必要的
- 設計趨勢會變，設計原則不變
