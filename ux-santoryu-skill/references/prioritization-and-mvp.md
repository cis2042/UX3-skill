# 需求取捨、MVP 與商業模式

> 當功能太多不知道先做什麼、需要規劃 MVP、或思考產品如何獲利時，使用這份指引。

---

## 1. 需求排序與取捨

### 為什麼要取捨？

- 使用者的需求永遠多於可用資源
- 一次滿足所有需求 = 什麼都做不好
- 取捨是設計師最重要的能力之一

### 取捨框架

**雙指標評估法**：

為每個需求在兩個維度評分：
- **使用者價值**（需求頻率 × 痛點強度 × 影響人數）
- **實現成本**（開發時間 × 技術難度 × 維護成本）

高價值低成本 → 優先做
高價值高成本 → 評估後決定
低價值低成本 → 有餘力再做
低價值高成本 → 不做

**商業流加權**：在用戶價值之外加入：
- 對公司策略目標的貢獻度
- 競品是否已有此功能
- 對利益關係人的影響

### 避免的陷阱

- 不要讓「聲音最大的人」主導優先順序
- 不要因為某個需求「容易做」就優先做（easy ≠ important）
- 不要把「老闆說的」自動排第一 — 用數據和框架說服

## 2. 產品藍圖（Product Roadmap）

### 角色

- 展示產品從現在到未來的發展路徑
- 對齊團隊共識：我們現在在哪、要去哪、怎麼去
- 溝通工具：讓工程、行銷、管理層都理解產品方向

### 規劃原則

1. **版本規劃**：將功能分配到不同版本（v1.0, v1.1, v2.0...）
2. **優先級驅動**：最核心的功能先上，驗證後再擴展
3. **保留彈性**：Roadmap 不是合約，是方向指引，要能因市場回饋調整
4. **里程碑明確**：每個版本要有明確的成功指標

## 3. 最小可行產品（MVP）

### 核心觀念

MVP 不是「做一個很爛的產品」，而是用**最小的資源**驗證**最關鍵的假設**。

### MVP 設計原則

1. **找到核心假設**：這個產品成功的前提是什麼？（例：用戶真的願意用手機訂外送嗎？）
2. **只做驗證假設所需的功能**：其他全部捨棄
3. **快速推出**：速度比完美重要
4. **收集數據**：上線後立刻追蹤關鍵指標
5. **根據結果決定下一步**：續做、轉向、還是放棄

### 成本思維

- 每個功能都有成本：開發、測試、維護、使用者學習
- 功能越多 ≠ 越好。過多功能 = 過高成本 + 過度複雜
- **先做能驗證市場的最小產品，而非最完美的產品**

## 4. 商業模式

### 核心問題

產品設計師必須回答：**我們打算怎麼跟客戶收錢？**

不同的收費方式 = 不同的產品設計

### 常見數位產品商業模式

- **付費下載**：一次買斷（需在下載前說服用戶付費）
- **免費增值（Freemium）**：基礎免費 + 進階付費（需設計功能付費邊界）
- **訂閱制**：按月/年付費（需持續提供足夠價值讓用戶續訂）
- **廣告模式**：免費使用，靠廣告收入（需大量用戶基數）
- **交易抽成**：平台撮合交易，抽取佣金（如外送、電商）
- **數據變現**：收集用戶行為數據，提供給廣告主

### 商業模式對 UX 的影響

每種模式都會影響產品設計：

| 模式 | 設計重點 |
|---|---|
| 免費增值 | 免費功能要有價值感，付費轉換要自然不突兀 |
| 訂閱制 | 持續提供新價值，降低退訂動機 |
| 廣告模式 | 廣告不能破壞核心體驗 |
| 交易抽成 | 撮合效率最重要，買賣雙方都要滿意 |

### 商業模式畫布（Business Model Canvas）

用一張畫布完整描述商業模式的九個要素：
1. 價值主張
2. 目標客群
3. 通路
4. 客戶關係
5. 收入來源
6. 關鍵資源
7. 關鍵活動
8. 關鍵合作夥伴
9. 成本結構

## 5. UX 品質與創新評估指標

### UX 品質怎麼衡量？

不能只說「體驗好/不好」— 要有具體指標。

**三層 UX 品質指標**：

1. **使用性（Usability）**：
   - 任務完成率
   - 操作錯誤率
   - 學習時間
   - 任務完成時間

2. **使用者滿意度**：
   - SUS（系統易用性量表）
   - NPS（淨推薦值）
   - 主觀滿意度評分

3. **商業指標**：
   - 轉換率
   - 留存率
   - 客單價
   - 客服量

### 創新評估

- 創新不是天馬行空，是在使用者心智模型的基礎上做有意義的突破
- 評估創新：是否解決了真實問題？使用者學習成本可接受嗎？市場時機對嗎？
