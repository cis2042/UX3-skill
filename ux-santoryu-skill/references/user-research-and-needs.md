# 使用者研究與需求分析

> 當你需要了解使用者、選擇研究方法、分析需求、或處理「設計師不是使用者」的困境時，使用這份指引。

---

## 1. 心智模型：UX 設計的根基

### 什麼是心智模型？

- **設計師的心智模型**（概念模型）：設計團隊對產品的想法與主張
- **使用者的心智模型**：使用者對產品的想像、認知與期待

**UX 設計的本質**：追求這兩個模型之間的最佳匹配。

### Don Norman 的心智模型框架

1. 設計團隊**透過 UX 研究**了解使用者 →
2. 設計團隊將理解注入產品設計 →
3. 使用者與產品互動，產生體驗 →
4. 收集使用者回饋 →
5. 繼續優化產品 →
6. 更接近使用者的期待

**這是一個迴圈，不是直線**。設計無法一次到位。

### 心智模型的應用

- **匹配原則**：產品的資訊架構、圖示、用語要符合使用者既有認知
- **創新空間**：UX 設計師是「心智模型的調酒師」— 拿捏人們不可動搖的期待，用低摩擦方式注入創新
- **案例**：Amazon Kindle 結合「書」和「平板」兩個已存在的心智模型，讓全新產品快速被接受

## 2. 以使用者為中心的設計（UCD）

### 核心原則

- 使用者從頭到尾參與設計過程
- 設計決策基於使用者的真實需求，而非設計師的主觀判斷
- 通過原型讓使用者實際體驗，而非只聽他們「說」想要什麼

### UCD 流程

1. 了解使用情境與需求
2. 制定使用者需求規格
3. 產出設計方案
4. 依據需求評估設計

這四步**持續循環迭代**，直到設計符合使用者需求。

### 關鍵心法

- 讓溝通透明：用 Prototype 讓客戶直接在頁面上留言
- 使用者說想要的 ≠ 使用者真正需要的 → 觀察行為比聽口述更可靠

## 3. 設計思考（Design Thinking）

### 五階段

1. **同理（Empathize）**：深入了解使用者的感受與處境
2. **定義（Define）**：從大量資訊中收斂出核心問題
3. **發想（Ideate）**：不設限的產生解決方案
4. **原型（Prototype）**：快速將想法具象化
5. **測試（Test）**：用原型驗證假設

### 設計思考的盲區

- 傳統設計思考高度依賴人的主觀判斷（如一人一張便利貼投票）
- 缺乏客觀數據支持 → 容易受團體動力、聲量大的人影響
- **運算式設計思考**：加入數據與運算技術，讓每個階段的決策更客觀
  - 同理階段用數據描繪用戶輪廓
  - 發想階段用數據驗證假設
  - 測試階段用 A/B 測試而非主觀判斷

## 4. 人因工程與產品設計

### 為什麼需要人因工程？

- 使用者之間存在個體差異：認知能力、年齡、經驗、文化背景
- 了解人的能力與限制，才能設計出「人用得了」的產品

### 關鍵認知原則

- **注意力有限**：一次能處理的資訊量有上限
- **記憶負荷**：減少使用者需要記住的東西（識別優於回憶）
- **認知負荷**：操作步驟越少、思考越少 = 越好的體驗
- **學習曲線**：利用已有的心智模型降低學習成本
- **容錯設計**：人會犯錯，系統要能防錯、容錯、幫助復原

### 不同使用者的設計考量

- 老人 vs 年輕人：認知速度、視力、手部精細動作
- 新手 vs 老手：引導需求、快捷操作
- 不同文化：閱讀方向、色彩意涵、圖示認知

## 5. 使用者研究方法

### 方法選擇原則

依據產品研發階段選擇合適的研究方法：

**探索階段**（了解市場與使用者）：
- 市場調查
- 使用者訪談（深度訪談）
- 田野觀察
- 問卷調查
- 焦點團體

**設計階段**（驗證設計方向）：
- 卡片分類法
- 原型測試
- A/B 測試
- 認知走查

**上線後**（持續優化）：
- 使用性測試
- 數據分析（行為數據）
- 客服回饋分析
- NPS / SUS 量表

### 重要觀念

- 方法是工具，不是目的。重要的是你**為什麼**在這個時間點用這個方法
- 不必所有方法都會，但要知道每種方法的適用時機
- 質化研究（理解 why）和量化研究（衡量 how much）互補使用

## 6. 使用者需求分析技術

### 需求擷取

從訪談、觀察、數據中提取使用者的：
- **表層需求**：使用者直接說出來的
- **深層需求**：使用者沒說出來但行為反映的
- **潛在需求**：使用者自己也沒意識到的

### 需求分析框架

- **User Story**：「作為 [角色]，我想要 [功能]，以便 [好處]」
- **客戶旅程地圖**：標記使用者與產品互動每個階段的情緒、行為、痛點
- **需求層次對應**：將需求對應到功能面、操作面、情感面

### 需求排序

需求總是多於資源。排序原則：

1. **使用者回饋指標**：需求的頻率、強度、影響範圍
2. **產品設計評估指標**：技術可行性、成本、與策略目標的對齊度
3. **Kano 模型思維**：
   - 基本需求（沒有會不滿，有了不加分）→ 必須做
   - 期望需求（做得好加分，做不好扣分）→ 重點投入
   - 興奮需求（有了驚喜，沒有也不扣分）→ 差異化機會

### 需求驗證

- 不要用「一人一張便利貼投票」做需求排序 — 不客觀
- 定義明確的評估指標，用多維度評分取代直覺投票
- 可使用加權評分矩陣：對每個需求在多個指標上評分，加權計算
